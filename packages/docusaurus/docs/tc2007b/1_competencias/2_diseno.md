# STC0203. Diseño de componentes de software
#### Nivel de la competencia C

## Evidencia

- Aplicación de diseño de software
- Evidencia de medición de los requisitos no funcionales
- Uso de las herramientas vistas en clase

| Nivel                                                                                    | Puntos |
| ---------------------------------------------------------------------------------------- | ------ |
| [Incipiente B](#incipiente-b--30pts)                                                     | -30    |

## Incipiente B (-30pts)

Hay desviaciones en el componente actitudinal de la competencia.

- La toma de decisiones dentro del proyecto lleva a riesgos su proyecto.
- Su actitud con su equipo de trabajo lleva a conflictos y no a soluciones.
- Existe ausencia dentro de su participación con el equipo de trabajo pese a que si cumple con sus actividades y entregables.
- No cumple con sus actividades y entregables.
- La comunicación que realiza tiene un impacto negativa (socio formador, compañeros y profesores)

## No hay evidencia o no se presentó - Incipiente B (0%)

Cualquiera de las siguientes es motivo de asignación directa de este nivel de dominio de la competencia al momento de su evaluación:

- No presentó evidencia o algún miembro del comité evaluador no tuvo acceso a ella previo a la evaluación.
- No se presentó el alumno o llegó tarde a la entrevista de evaluación. 
- No se observa desarrollo de la competencia.
- No conoce (haber leído previamente) lo que abarca la competencia.
- No hay evidencia de conocimiento del diseño de software

## Incipiente A (15%)

- Conoce lo que es el diseño pero no hay evidencia de identificación y/o aplicación del diseño de software
- No utiliza componentes de diseño de software para plantear los requerimientos de un software.
- No hay evidencia de ejecución de las etapas de diseño antes de pasar a la implementación
- No es capaz de diseñar la arquitectura de cualquier sistema computacional
- No hay evidencia de entender y/o conocer los requisitos no funcionales y su importancia en el diseño de software
- Hay evidencia que no pertenece a la etapa de diseño de software

## Básico (70%)

Conoce, identifica, describe de manera adecuada y medible requisitos no funcionales

Identifica de manera adecuada componentes de diseño de software que se utilizan para la construcción de un software, sin embargo no puede aplicarlos de manera adecuada a los requerimientos de un software.

Realiza el diseño de sistemas de software que requieren de interfaces de usuario, se muestra evidencia de diseño detallado previo a pasar a la etapa de implementación

## Sólido (85%)

Diseña componentes de software de acuerdo con las necesidades de un equipo de trabajo y de acuerdo con una especificación de requerimientos funcionales

Realiza el diseño de sistemas de software que requieren de interfaces de usuario y persistencia de datos, se muestra evidencia de diseño detallado y arquitectónico previo a pasar a la etapa de implementación

Diseña la arquitectura del sistema a desarrollar integrando servicios de externos.

## Destacado (100%)

Diseña componentes de software de acuerdo con las necesidades de un equipo de trabajo y de acuerdo con una especificación de requerimientos funcionales y no funcionales.

Realiza el diseño de sistemas de software que requieren de interfaces de usuario, persistencia de datos, acceso a recursos, servicios en red y hardware especializado. Muestra evidencia de ejecución de la etapa de diseño previo a la etapa de implementación.

Diseña componentes que permitan optimizar los recursos y la arquitectura del sistema a desarrollar.
