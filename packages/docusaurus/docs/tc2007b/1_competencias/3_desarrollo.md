# STC0204: Desarrollo de componentes de software

## Datos Generales

- Nivel de Competencia: **B**
- Para alcanzar un nivel de dominio se requiere haber cumplido con **TODO** el OKR de nivel de dominio que lo conforma.
- Para alcanzar el siguiente nivel de dominio se requiere haber alcanzado el previo.
- Para las asignaciones directas cualquiera de las que lo conforman es motivo de asignación directa del nivel correspondiente de dominio de la competencia.
- Se califican **historias de usuario** no Pull Request, podrás tener 1000 PR, pero si no hay historia de usuario completa y trazable no contará.
- El Pair Programming podrá ser utilizado para facilitar su trabajo, sin embargo para la calificación solo contará la **historia de usuario** para 1 sola persona, en caso de realizar esta técnica deberán decidir quien es el lead programmer y el responsable de la misma.
- Bajo ninguna circunstancia es justificable que el proyecto no tenga las suficientes historias de usuario para todos los miembros del equipo, si hacen falta significa que no están desglosando bien el proyecto o en su defecto no están añadiendo suficiente valor al mismo.

## Incipiente B (-30pts)

Hay desviaciones en el componente actitudinal de la competencia. 

La ejecución de una o más de las siguientes acciones son base para asignar el nivel:
- No escucha las recomendaciones sobre desarrollo y riesgos
- Existe evidencia de desarrollo de laboratorios pero no existe participación o evidencia de desarrollo en el proyecto.
- Existe evidencia de desarrollo en el proyecto pero no existe evidencia de desarrollo de laboratorios.
- Deja todo al final incluso si existe firma del cliente donde se acepte el alcance tomado en tiempo y forma.
- Modifica el plan de desarrollo no siendo aprobado por el correspondiente product owner o siendo este mismo haciendo cambios sin notificar a sus compañeros siguiendo la metodología establecida por los miembros de toda la tribu.
- Conoce pero no aplica las estrategia de arquitectura de desarrollo de componentes de software o solamente esta en contra de ellos sin proponer planes alternativos o recursos que den beneficio a sus compañeros.
- Genera más obstáculos o conflictos de los que resuelve en los proyectos desarrollados por la tribu.
- Utiliza código el cual no sabe o entiende su uso o funcionamiento independientemente su procedencia.
- **Hay una sobre autoevaluación en la competencia sin reflexión y/o evidencia de soporte.**

> Nota: Comportamientos no contemplados en este escrito serán sujetos a la evaluación por parte del equipo de profesores según sea el caso.



## No hay evidencia o no se presentó - Incipiente B (0%)

Cualquiera de las siguientes es motivo de asignación directa de este nivel de dominio de la competencia al momento de su evaluación:

- No presentó evidencia o algún miembro del comité evaluador no tuvo acceso a ella previo a la evaluación.
- No se presentó el alumno o llegó tarde a la entrevista de evaluación. 
- No se observa desarrollo de la competencia.
- No conoce (haber leído previamente) lo que abarca la competencia.
- No conoce sobre la implementación de su proyecto.
- No hay evidencia ordenada de la implementación de **historias de usuario**, es decir, puede tener código o commits en el proyecto pero estas no son fuente completa del desarrollo de al menos 1 historia de usuario, el historial de commits es tan largo que no sigue un proceso de integración continua basado en los estándares del proyecto y los commits no representan un historial de desarrollo sobre las 10 semanas que dura el proyecto y solo son durante los últimos días o la última semana a la evaluación.
- Se entregan Pull Requests solamente y no historias de usuario completas.

## Incipiente A (15%)

### Asignaciones directas
- No conoce o entiende parcialmente el significado de historia de usuario.
- Desarrolla 1 **historia de usuario** codifica (existe el/los pull request aprobado y mergeado a master o develop asociado a la historia de usuario) del proyecto y está no tiene evidencia de cubrir el ciclo de desarrollo de software (análisis, diseño, pruebas, validación e implantación) así como la definición de ready y done definida por el equipo de desarrollo.
- La historia de usuario no se puede asociar al uso de componentes de software como arquitectura, patrones de diseño o de arquitectura. Al preguntarle sobre las arquitecturas del proyecto solo menciona MVC.
- Las historias de usuario desarrolladas están documentadas de manera simple, esto es solo contienen comentarios sin seguir un estándar de documentación de código.

### OKR de nivel de dominio

#### Objetivo
El alumno no ha desarrollado componentes diseñados de un sistema computacional. No desarrolla front-end o backend y no presenta planeación en historias de usuario por su cuenta.

#### Key Result
- [ ] Su calificación en el examen final de desarrollo móvil es igual o menor a 35.

## Básico (70%)

### Asignaciones directas
- Aplica la teoría de una historia de usuario pero no añade valor al proyecto.
- Desarrolla 2 **historias de usuario** del sistema codifica (existe el/los pull request aprobado y mergeado a master asociado a la historia de usuario) del proyecto y está tiene evidencia incompleta de cubrir el ciclo de desarrollo de software (análisis, diseño, pruebas, validación e implantación) así como la definición de ready y done definida por el equipo de desarrollo.
- La historia de usuario se puede asociar al uso de componentes de software como arquitectura, patrones de diseño o de arquitectura.
- Las historias de usuario desarrolladas están documentadas siguiendo un estándar de documentación de código.

### OKR de nivel de dominio

#### Objetivo

El alumno ha desarrollado componentes diseñados de un sistema computacional. Desarrolla front-end o backend para desarrollo web y presenta planeación incompleta en historias de usuario por su cuenta.

#### Key Result
- [ ] Su calificación en el examen final de desarrollo móvil es igual o menor a 70.

## Sólido (85%)

### Asignaciones directas
- Desarrolla 3 **historias de usuario** del sistema codifica (existe el/los pull request aprobado y mergeado a master asociado a la historia de usuario) del proyecto y está tiene evidencia incompleta de cubrir el ciclo de desarrollo de software (análisis, diseño, pruebas, validación e implantación) así como la definición de ready y done definida por el equipo de desarrollo.
- La evidencia del desarrollo de historias de usuario tiene una trazabilidad complicada de seguir de más de 1 ligas de seguimiento (links a evidencias).
- La historia de usuario se puede asociar al uso de componentes de software como arquitectura, patrones de diseño o de arquitectura, pero el alumno aún no entiende para que el uso de diferentes arquitecturas, sus componentes y como aplicarlos en un proyecto de desarrollo móvil.
- Las historias de usuario desarrolladas están documentadas siguiendo un estándar de documentación de código.

### OKR de nivel de dominio

#### Objetivo

El alumno ha desarrollado los componentes diseñados de un sistema computacional. Desarrolla front-end o backend para desarrollo y presenta planeación completa en historias de usuario por su cuenta.

#### Key Result
- [ ] Su calificación en el examen final de desarrollo móvil es igual o menor a 99.

## Destacado (100%)

### OKR de nivel de dominio

#### Objetivo
El alumno ha desarrollado la capacidad de compartir lo aprendido en el desarrollo de un sistema computacional. Utiliza documentos, diagramas y estándares para facilitar el trabajo de su equipo así como los utiliza en el momento preciso.

#### Key Result
- [ ] Desarrolla 3 o más historias de usuario del sistema codifica (existe el pull request aprobado y mergeado a master asociado a la historia de usuario) del proyecto y está tiene evidencia completa de cubrir el ciclo de desarrollo de software (análisis, diseño, pruebas, validación e implantación) así como la definición de ready y done definida por el equipo de desarrollo.
- [ ] La evidencia del desarrollo de historias de usuario tiene una trazabilidad fácil de seguir desde el mismo documento a máx. 1 ligas de seguimiento (links a evidencias).
- [ ] El alumno puede identificar fácilmente los beneficios de las diferentes arquitecturas de desarrollo móvil y cuando aplicar una u otra en un proyecto, así como sus componentes.
- [ ] Su calificación en el examen final de desarrollo móvil es 100.
