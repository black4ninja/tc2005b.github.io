0Resumen ejecutivo

Contenidos es el módulo CMS autoadministrable que reemplaza a Docusaurus en la plataforma. Conserva lo que Docusaurus hace bien —árbol de navegación, experiencia de lectura, Markdown de primera— y corrige lo que no puede dar: control de acceso real por request, edición desde la web y contenido vivo en base de datos.

🔒 Seguro por diseño
El contenido nunca viaja en un bundle estático. Cada página, índice, búsqueda y asset se entrega desde el servidor con autorización por request. Acceso denegado = 404.
✏️ Editable desde web
Editor tipo VS Code (CodeMirror 6): Markdown con resaltado, preview en vivo, pegado de imágenes, admonitions, tablas… y modo HTML crudo para demos interactivas.
🗄️ BD única fuente de verdad
Parse/MongoDB con versionado por página (historial y restaurar). Sin sync con archivos: la migración importa el contenido Docusaurus una sola vez, al 100%.

Por qué reemplazar Docusaurus

Fuga actual (Docusaurus estático + gate)Cómo la cierra Contenidos
El navbar muestra todas las materias a cualquier usuario (revela estructura ajena). La navegación se calcula por usuario en el servidor: solo ve colecciones permitidas.
El SPA precarga bundles/JSON de páginas de otras materias desde el cliente. No hay bundle de contenido: cada página llega por API autorizada; lo no permitido no existe (404).
La búsqueda indexa globalmente y sugiere títulos/contenido de otras materias. Búsqueda server-side con scope por permisos: consulta solo colecciones del usuario.
Assets (PDF/ZIP/img) en /docs/img, /docs/ppts… son públicos con URL directa. Assets en storage referenciado desde BD, servidos por endpoint gated con la misma autorización.

Decisiones clave (resueltas con el usuario)

  • Migración total: Docusaurus se depreca. El importador preserva el 100% del contenido y la navegación por índice; el chrome (sidebar/topbar/footer) se rediseña.
  • Edición solo web: sin sync con archivos — la BD manda. A cambio, el editor debe estar a la altura de VS Code.
  • Administración fuera de los grupos: el CMS es una sección propia del admin; los grupos solo consumen vía asignación de colecciones (hereda el patrón Grupo→materia/docusaurus[]).
  • Página = .md o .html crudo: el HTML se sirve sandboxeado para demos interactivas.

1Modelo de contenido

Cuatro clases Parse nuevas siguiendo las convenciones del sistema: subclases de Parse.Object vía BaseModel, pointers para toda relación (nunca strings), soft-delete con exists/active, y toSafeJSON() para exponer al cliente.

Coleccion — equivale a una instancia Docusaurus
  • nombre string — "Construcción de software…"
  • slug string único — "tc2005b" (kebab, ruta)
  • clave string — "TC2005B" (nomenclatura CLAVE — Nombre)
  • descripcion string?
  • icono string — Material Icon
  • materia → Pointer<Materia>? — vínculo opcional
  • publicada boolean — borrador vs visible
  • active / exists boolean — patrón BaseModel
Documento — una página del árbol
  • coleccion → Pointer<Coleccion>
  • padre → Pointer<Documento>? — null = raíz
  • titulo string
  • slug string — único por (coleccion, padre)
  • tipo 'md' | 'html' | 'categoria'
  • orden number — = sidebar_position
  • plantilla 'laboratorio'|'lectura'|'temario'|null
  • version → Pointer<DocumentoVersion> — la publicada
  • borrador → Pointer<DocumentoVersion>? — si difiere
  • publicado boolean
DocumentoVersion — historial inmutable
  • documento → Pointer<Documento>
  • numero number — incremental por documento
  • cuerpo string — fuente MD o HTML
  • cuerpoHtml string — render cacheado (pipeline unified)
  • autor → Pointer<AppUser>
  • mensaje string? — "qué cambió" (opcional)
Recurso — asset adjunto (img/pdf/zip)
  • coleccion → Pointer<Coleccion>
  • documento → Pointer<Documento>? — o global de colección
  • nombre string — nombre de archivo visible
  • archivo Parse.File — storage vía files adapter
  • mime / bytes string / number
  • subidoPor → Pointer<AppUser>
Asignación a grupos — extiende el modelo actual Grupo gana colecciones: Pointer<Coleccion>[] (array de pointers, no slugs — corrige la deuda del actual docusaurus: string[]). El acceso de un alumno = unión de las colecciones de sus grupos activos, igual que hoy con materia + docusaurus. Durante la transición conviven ambos campos; al deprecar Docusaurus, docusaurus[] se elimina.

El árbol reemplaza al sidebar autogenerado

Docusaurus deriva el sidebar de carpetas + _category_.json + sidebar_position. Aquí el árbol es el dato: Documento.padre + orden. Una categoria es un Documento sin cuerpo que agrupa hijos (equivalente a carpeta con _category_.json); si se le da cuerpo, actúa como su página índice (intro.md). Reordenar = drag & drop que reescribe orden — sin renombrar prefijos 1_, 2_… como hoy.

Versionado

  • Cada guardado publica o acumula en borrador; "Publicar" congela una DocumentoVersion nueva e incrementa numero.
  • El historial lista versiones con autor/fecha/mensaje; restaurar = crear una versión nueva con el cuerpo de la vieja (nunca se muta el pasado).
  • cuerpoHtml se renderiza al publicar (server-side, pipeline de la §3) — servir una página es leer un campo, no re-renderizar.
Relación con la clase Pagina existente La actual Pagina (bloques tipados encabezado/objetivos/…, scoped a un grupo) resuelve otro problema: páginas operativas del grupo. Se mantiene intacta. Contenidos es documentación de estudio por colección. Si a futuro conviene converger, una plantilla de Documento puede absorber esos bloques — decisión explícitamente fuera de este alcance.

2Seguridad por diseño

La regla es una sola: ningún byte de contenido sale del servidor sin pasar por autorización de ese request. El shell de la aplicación (React) es público; el contenido, nunca.

Flujo de un request de lectura

GET /api/contenidos/tc2005b/backend/lab1cookie httpOnly ó header x-session-token (getSessionToken ya existente)
↓
identifyUserAppSession → AppUser (middleware actual)
resolver permisoscoleccionesPermitidas(user) — cache 60s por usuario, patrón materia.service
↓
¿slug ∈ permitidas?NO → 404 (sin filtrar existencia) · SÍ → query Documento publicado
↓
respuesta JSON{ titulo, cuerpoHtml, toc, prev, next, breadcrumb } — solo lo de ESTA página

Superficies y su gate

SuperficieEndpointQué autoriza
Índice / navegaciónGET /api/contenidos/:col/arbolDevuelve el árbol solo si la colección está permitida; el "navbar" del visor lista únicamente GET /api/me/colecciones.
PáginaGET /api/contenidos/:col/pagina/*pathColección permitida + documento publicado. 404 en cualquier otro caso.
BúsquedaGET /api/contenidos/busqueda?q=…Índice de texto MongoDB filtrado por coleccion ∈ permitidas — imposible sugerir contenido ajeno.

HTML crudoGET /api/contenidos/:col/html/*pathMisma autorización; se sirve para consumo solo dentro de iframe sandbox (§3).

Caches e invalidación (patrón ya probado en materia.service)

  • Por usuario (60s): set de colecciones permitidas — se invalida al cambiar asignaciones de grupo, enrollment o colecciones (mismos hooks que hoy invalidan allowedCache).
  • Global (5min): mapa slug→coleccionId publicadas — se invalida en CRUD de colecciones.
  • Render: cuerpoHtml se cachea en la versión (BD), no en memoria: publicar invalida por construcción.

El mismo pipeline renderiza el preview del editor y el cuerpoHtml publicado — cero divergencia entre lo que se edita y lo que se sirve.

Política del HTML crudo (tipo html)

  • El HTML de una página html nunca se inyecta en el DOM de la app: se sirve por su endpoint gated y se muestra en <iframe sandbox="allow-scripts"> — sin allow-same-origin, el iframe corre en origen opaco: no ve cookies (que además son httpOnly), ni localStorage, ni la sesión.
  • El endpoint lo entrega con CSP propia (default-src 'self' 'unsafe-inline'; frame-ancestors groups.meeplab.com) para impedir que cargue recursos de terceros no deseados o sea embebido fuera.
  • El Markdown, en cambio, pasa por rehype-sanitize (allowlist) — el HTML inline peligroso se elimina en el render.
  • Solo admin/profesor crean páginas html; el riesgo residual (XSS dentro del iframe aislado) queda confinado a contenido que el propio docente autoría.

Redirects

  • /docs/<slug>/<path> → /contenidos/<slug>/<path> — 301 en Express (tabla generada por el importador, incluye los redirects legacy que hoy resuelve el plugin client-redirects).
  • Los enlaces en BD (Pagina registradas) y en datos (data/labs/*) se migran con un script tipo migrate-paginas-docs-url.ts, como en la US de simplificación de URLs.

Plan de deprecación de packages/docusaurus

FaseEstadoSalida
1 · ParaleloCMS activo con contenido importado; Docusaurus sigue sirviendo /docs.Validación de paridad página por página (checklist del importador).
2 · CorteFreeze de edición en Docusaurus; 301 /docs/* → /contenidos/*; links de la plataforma actualizados.Docusaurus ya no recibe tráfico directo.

Regla de la fase 1 No se depreca nada hasta comprobar en producción que cada página importada abre, navega y descarga sus recursos igual que su original. El importador emite un reporte de paridad (páginas, enlaces reescritos, enlaces rotos preexistentes, assets migrados).
